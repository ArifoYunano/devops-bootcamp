1. Create a Security Group to allow HTTP and HTTPS traffic to an Nginx web server.

Create Security Group: devops-sg
Attach it to devops-vpc
Allow:
HTTP (Port 80) from 0.0.0.0/0
HTTPS (Port 443) from 0.0.0.0/0
Keep outbound rules as All traffic (default)
Purpose: This Security Group enables public web access to an Nginx server while allowing the instance to communicate outbound as needed.

2. Launch an Ubuntu EC2 instance, attach the appropriate Security Group and IAM Role, then assign an Elastic IP for a permanent public IP address.

Launch an Ubuntu 24.04 EC2 instance (t3.micro).
Deploy it in the devops-vpc using a Public Subnet.
Attach the devops-sg Security Group.
Launch without a key pair.
Assign the IAM Instance Profile EC2-SSM-Role.
Associate an Elastic IP with the EC2 instance for a permanent public IP address.
Purpose: This setup creates a publicly accessible EC2 web server that can be managed securely through AWS Systems Manager (SSM) and retains a fixed public IP using an Elastic IP.

3. Connect to the EC2 instance using AWS Systems Manager (SSM), install Nginx, verify the service is running, and access the web server through its Elastic IP.
Connect to EC2 using AWS Session Manager (SSM).
Update the package repository. (sudo apt update)
Install Nginx and dnsutils. (sudo apt install -y nginx dnsutils)
Verify Nginx is running using systemctl status nginx.
Ensure HTTP (80) and HTTPS (443) are allowed in the devops-sg Security Group.
Access the server using:
http://<Elastic-IP>
Result: Your EC2 instance is now running an Nginx web server and is publicly accessible through its Elastic IP address.

4. Configure a DNS A Record so that your domain (or subdomain) points to your EC2 instance using its Elastic IP.

Go to DNS → Records → Add Record.
Create an A Record:
Type: A
Name: web
IPv4 Address: <Elastic-IP>
Set Proxy Status to DNS Only (Grey Cloud).
Save the record.
Verify by visiting:
http://web.<your-domain>
Result: Your domain (or subdomain) now resolves to your EC2 instance via its Elastic IP, allowing users to access the Nginx web server using a domain name instead of an IP address.

5. Cloudflare Proxy Status & IP Visibility

Before change — While the web DNS record is still grey (DNS Only), run dig web.domain +short from the EC2 server via SSM session → it returns the real Elastic IP.
Switch to orange — In Cloudflare dashboard, go to DNS → web record → click the cloud icon → Save to enable proxying.
After change — Run dig web.domain +short again → it now returns a Cloudflare IP, meaning the Elastic IP is hidden.
Verify from Windows laptop — Run nslookup web.domain → result is the same Cloudflare IP, confirming the proxy is active globally.

When proxy is grey, anyone can discover your server's real IP. Switching to orange routes all traffic through Cloudflare's edge network, concealing the origin and enabling DDoS protection and caching.

6. This note explains a situation where the origin server supports only HTTP, and how Cloudflare Flexible SSL can be used so the website remains accessible securely from the user’s side.

Before: Nginx on the origin only uses port 80 and has no SSL certificate, so Cloudflare cannot connect to the origin over HTTPS; this may cause errors such as 522 or an insecure connection.
After selecting Flexible: The connection from the browser to Cloudflare uses HTTPS, while the connection from Cloudflare to the origin uses HTTP.
As a result, the site can still appear secure to the user even though the origin server does not yet have its own SSL certificate.

Setting: Cloudflare Dashboard → SSL/TLS → Overview → Configure → Flexible

Flexible SSL is suitable for temporary setups or learning labs, but for production use, it is better to use Full or Full (Strict) with a valid certificate on the origin server for stronger end-to-end security.

7. This practical shows the basic steps to create a Cloudflare Tunnel through the Zero Trust dashboard so a private EC2 server can be connected securely without directly exposing it to the internet.

Open the Cloudflare Dashboard and go to Zero Trust; for first-time setup, create a team name and use the Free plan.
Go to Networks → Overview → Manage Tunnels → Create a tunnel → Cloudflared to start creating the tunnel.
Enter the tunnel name and select the operating system, shown here as Debian 64-bit, then save the configuration.
Copy the provided connector installation command and run it on the private EC2 instance to connect that server to Cloudflare.

A Cloudflare Tunnel allows services running on a private server to be published securely through Cloudflare without opening inbound public access directly to the EC2 machine.

8. This practical explains how to configure a subdomain to point to a local localhost service through Cloudflare Tunnel, so the service can be accessed publicly via a web address.

In the selected tunnel, go to Public Hostname and choose Add a public hostname.
Set the subdomain and domain name, for example subdomain.domain.com.
Point it to the local service by choosing Service: HTTP and entering URL: localhost:80, then save the configuration.
After saving, open https://subdomain.domain.com and the local web service will be published through Cloudflare Tunnel.

This setup allows a service running only on localhost to be reached externally using a subdomain, without directly exposing the server to inbound public traffic.
